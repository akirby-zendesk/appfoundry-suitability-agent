# Real-World Assessment: Duplicate Finder Dashboard

This is the actual assessment that led to the creation of this agent. It demonstrates a comprehensive analysis of a production Flask application.

## Project Background

**Duplicate Finder Dashboard** - Python + Flask dashboard detecting suspicious opportunity re-attribution in Salesforce. 

- Queries Snowflake SCD2 tables
- Scores pairs across 9 signals
- LLM-powered investigation via Zendesk AI Gateway
- Server-Side Events for streaming
- Follow-up chat with SQL agent

**Original Deployment**: Run locally via `python app.py` on port 5001

**Goal**: Deploy to App Foundry for internal team access

## Key Findings

### Compatibility Score: MEDIUM (70% Ready)

The application was already 70% migrated with an `appfoundry/` subdirectory containing:
- Flask app with Pomerium JWT authentication ✅
- PostgreSQL models (SQLAlchemy) ✅
- File storage replaced with database ✅
- Gunicorn setup ✅

### 4 Critical Blockers Identified

1. **Snowflake `externalbrowser` Authentication**
   - Problem: No browser in Cloud Run container
   - Solution: Service account with JWT key-pair auth
   - External dependency: Requires data platform team approval

2. **Stateful Investigation Context**
   - Problem: Follow-up chat stored in `_investigation_contexts` dict
   - Solution: PostgreSQL table with JSON storage + 24-hour TTL
   - Implementation: ~2 hours

3. **Background Detection Job**
   - Problem: `detector.py` is a CLI script
   - Solution: HTTP endpoint + Cloud Scheduler trigger
   - Implementation: ~1 hour

4. **SSE Streaming Reliability**
   - Problem: Long-lived connections + container restarts
   - Solution: Heartbeat messages + partial progress storage
   - Implementation: ~2 hours

## Remediation Plan

**Phase 1: Critical Fixes (1-2 days)**
- Request Snowflake service account
- Add InvestigationContext database model
- Create detection job HTTP endpoint
- Improve SSE reliability

**Phase 2: Optimization (1 day)**
- Add connection pooling (PostgreSQL + Snowflake)
- Configure Cloud Scheduler
- Add health checks

**Timeline**: 1-2 weeks (including external service account approval)

## Key Architectural Changes

### Before (Local Development)
```python
# File storage
with open('results/latest.json', 'w') as f:
    json.dump(results, f)

# Snowflake auth
SNOWFLAKE = {"authenticator": "externalbrowser"}

# In-memory context
_investigation_contexts = {}
```

### After (Cloud Run)
```python
# Database storage
run = DetectorRun(results_count=len(results))
db.session.add(run)
db.session.commit()

# Service account auth
SNOWFLAKE = {
    "authenticator": "SNOWFLAKE_JWT",
    "private_key": os.environ["SNOWFLAKE_PRIVATE_KEY"]
}

# Database context
class InvestigationContext(Base):
    context_data = Column(JSON)
    expires_at = Column(DateTime)
```

## Lessons Learned

### What Made This Assessment Effective

1. **Started with existing migration work** - 70% done meant clear path forward
2. **Identified external dependencies early** - Snowflake service account blocked deployment
3. **Prioritized by impact** - Critical blockers first, optimizations second
4. **Provided code examples** - Every fix had before/after code
5. **Realistic timeline** - Accounted for external approval process

### Common Anti-Patterns Found

- ❌ Browser-based authentication (Snowflake externalbrowser)
- ❌ File-based storage (results/latest.json)
- ❌ In-memory state (investigation context dict)
- ✅ Background job pattern (detector.py CLI → HTTP endpoint)

### Why This App Was a Good Fit

- Internal tool (Pomerium auth perfect fit)
- Database-backed (PostgreSQL available)
- Request-driven UI (dashboard pattern)
- Periodic batch job (Cloud Scheduler works)
- LLM integration (API-based, works great)
